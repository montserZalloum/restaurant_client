# وثيقة متطلبات المنتج (PRD)
## المرحلة الأولى: الاختبار المحلي على جهاز ويندوز

---

## 1. نظرة عامة

### الهدف من هذه المرحلة
إثبات إنه فكرة اعتراض بيانات الطباعة من الشبكة وتمريرها للطابعة الحقيقية مع استخراج رقم الطلب **تشتغل تقنياً**، قبل صرف أي مال على هاردوير حقيقي أو الذهاب لأي مطعم.

### نطاق هذه المرحلة
- محاكاة كاملة على جهاز ويندوز واحد (لابتوب أو ديسكتوب)
- ثلاث مكونات وهمية تشتغل على نفس الجهاز
- بدون أي طابعة أو كاشير حقيقي

### خارج النطاق (للمراحل القادمة)
- التركيب على جهاز كاشير حقيقي
- الاتصال بطابعة حرارية حقيقية
- السيرفر السحابي
- لوحة الإعدادات
- التركيب كـ Windows Service

- **الكاشير الوهمي** يبعت إلى: `127.0.0.1:9100`

## 3. المكونات المطلوبة

### المكون 1: الكاشير الوهمي (Fake Cashier)

**الغرض:** محاكاة برنامج كاشير حقيقي يبعت أوامر طباعة عبر TCP.

**المتطلبات الوظيفية:**
- يفتح اتصال TCP على `127.0.0.1:9100`
- يبعت بيانات ESC/POS تمثل فاتورة كاملة
- يحتوي رقم طلب واضح (مثلاً سطر فيه "Order #123" أو "رقم الطلب: 123")
- يحتوي بعض الأصناف بأسعار وهمية
- بعد الإرسال يقفل الاتصال

**التحكم بالاستخدام:**
- يشتغل من Terminal بأمر `node fake-cashier.js`
- كل ما يتشغل، يبعت طلب جديد برقم متزايد
- يطبع على الـ console معلومات إيش بعت

**شكل البيانات المطلوبة (مثال نص الفاتورة):**
```
================================
       مطعم الاختبار
       فرع الجامعة
================================
رقم الطلب: 47
التاريخ: 26/04/2026
الوقت: 14:32

شاورما عربي         3.50
كولا                 1.00
بطاطا                 1.50
--------------------------------
المجموع              6.00
================================
       شكراً لزيارتكم
================================
```

**ملاحظة تقنية:** نبدأ بإرسال نص عادي مع بعض أوامر ESC/POS الأساسية (تهيئة، قص الورق). لا نحتاج كل بروتوكول ESC/POS بهذه المرحلة.

---

### المكون 2: البرنامج المحلي (Local Agent)

**الغرض:** هذا هو المنتج الفعلي الذي سيتم تطويره. يعترض بيانات الطباعة، يمررها للطابعة، ويستخرج رقم الطلب.

**المتطلبات الوظيفية:**

#### أ. استقبال البيانات
- يفتح TCP server على `127.0.0.1:9100`
- يقبل اتصالات متعددة (لكن متتالية بهذه المرحلة)
- يقرأ كامل البيانات حتى يقفل الكاشير الاتصال أو يصل لنهاية الفاتورة

#### ب. التمرير الفوري للطابعة (الأولوية القصوى)
- بمجرد استقبال أول جزء من البيانات، يفتح اتصال TCP إلى `127.0.0.1:9101` (الطابعة الوهمية)
- يمرر البيانات كما هي، بدون أي معالجة أو تأخير
- وقت التمرير الإجمالي يجب أن يكون أقل من 50ms
- إذا فشل الاتصال بالطابعة، يحاول 3 مرات بفواصل 200ms

#### ج. حفظ البيانات الخام
- بعد التمرير، يحفظ البيانات الخام في قاعدة بيانات SQLite محلية
- يحفظ: المعرّف، الوقت، البيانات الخام (نص + bytes), حالة المعالجة

#### د. استخراج رقم الطلب
- بعد الحفظ، يحاول استخراج رقم الطلب من النص
- يستخدم regex بسيط للبحث عن أنماط مثل:
  - `رقم الطلب:\s*(\d+)`
  - `Order\s*#\s*(\d+)`
  - `Order\s*Number:\s*(\d+)`
- إذا فشل الاستخراج، يستخدم رقم تسلسلي داخلي

#### هـ. إرسال للسيرفر
- يرسل رقم الطلب + الوقت إلى السيرفر المحلي على `http://127.0.0.1:3000/api/orders`
- إذا فشل الإرسال، يحفظ في قائمة انتظار ويعيد المحاولة بعد 5 ثواني

#### و. التسجيل (Logging)
- يطبع على الـ console كل خطوة:
  - "وصلت بيانات بحجم X bytes"
  - "تم التمرير للطابعة في Yms"
  - "تم استخراج رقم الطلب: Z"
  - "تم الإرسال للسيرفر بنجاح"

**المتطلبات غير الوظيفية:**
- مكتوب بـ Node.js + TypeScript
- بدون أي واجهة رسومية، يشتغل بـ Terminal فقط
- يقدر يشتغل لساعات بدون مشاكل ذاكرة
- يتعامل مع الأخطاء بدون ما يقع

### المكون 3: الطابعة الوهمية

**الغرض:** محاكاة طابعة حرارية حقيقية تستقبل أوامر ESC/POS.

**الحل المقترح: استخدام escpresso**
- مشروع مفتوح المصدر مجاني
- يشغل TCP server على البورت 9100 (نغيره لـ 9101 بحالتنا)
- يعرض الفاتورة بـ GUI مرئي بشكل لحظي
- متوفر على: https://github.com/jflaflamme/escpresso

**خطوات التشغيل:**
1. تنزيل escpresso أو بناءه من المصدر
2. تشغيله على البورت 9101
3. التأكد من ظهور الـ GUI

**البديل البسيط لو escpresso ما اشتغل:**
كتابة سكريبت Node.js بسيط جداً يفتح TCP server على 9101 ويطبع كل البيانات اللي وصلت على الـ console (بدون GUI).

---

### المكون 4: السيرفر المحلي + شاشة العرض

**الغرض:** استقبال أرقام الطلبات من البرنامج المحلي وعرضها على شاشة ويب.

**المتطلبات الوظيفية:**

#### أ. السيرفر (Backend)
- مكتوب بـ Node.js + Fastify (أو Express لو تفضل)
- يستمع على `127.0.0.1:3000`
- نقطتين رئيسيتين:
  - `POST /api/orders` - استقبال طلب جديد من البرنامج المحلي
  - `GET /` - تقديم صفحة الشاشة
  - `GET /staff` - تقديم صفحة موظف الاستلام
- يحفظ الطلبات في الذاكرة (Array بسيط) - بدون قاعدة بيانات بهذه المرحلة
- WebSocket للتحديث اللحظي

#### ب. شاشة العرض (Display)
- صفحة HTML واحدة
- مقسومة لقسمين كبار:
  - **يسار: قيد التحضير** (لون رمادي/أزرق)
  - **يمين: جاهز للاستلام** (لون أخضر)
- كل قسم يعرض الأرقام بحجم كبير جداً (200px+)
- يتحدث لحظياً عبر WebSocket
- صوت تنبيه عند انتقال طلب لقسم "جاهز"

#### ج. صفحة موظف الاستلام
- صفحة HTML بسيطة يفتحها موبايل الموظف
- تعرض قائمة الطلبات قيد التحضير
- زر كبير لكل طلب: "جاهز"
- زر آخر بعد ما يصير جاهز: "تم التسليم"

## 4. السيناريو الكامل للاختبار

### السيناريو 1: التدفق السعيد (Happy Path)

**الخطوات:**
1. شغّل الطابعة الوهمية على بورت 9101
2. شغّل السيرفر المحلي + الشاشة على بورت 3000
3. افتح صفحة الشاشة في المتصفح: `http://localhost:3000`
4. افتح صفحة موظف الاستلام في تبويب آخر: `http://localhost:3000/staff`
5. شغّل البرنامج المحلي (يبدأ يستمع على 9100)
6. شغّل الكاشير الوهمي

**النتيجة المتوقعة:**
- الطابعة الوهمية تعرض الفاتورة كاملة
- البرنامج المحلي يطبع رسائل النجاح في الـ console
- رقم الطلب يظهر في "قيد التحضير" على الشاشة
- نفس الرقم يظهر في صفحة الموظف
- لما تضغط "جاهز" في صفحة الموظف، الرقم ينتقل لـ "جاهز للاستلام" مع صوت تنبيه

### السيناريو 2: عدة طلبات متتالية
- شغّل الكاشير الوهمي 5 مرات بسرعة
- النتيجة: 5 أرقام تظهر على الشاشة بنفس الترتيب

### السيناريو 3: الطابعة مفصولة
- اقفل الطابعة الوهمية
- شغّل الكاشير الوهمي
- النتيجة المتوقعة: البرنامج يحاول 3 مرات، يطبع رسالة فشل، لكن لا يقع. لما ترجع تشغل الطابعة، يكمل عمله طبيعي.

### السيناريو 4: السيرفر مقفل
- اقفل السيرفر المحلي
- شغّل الكاشير الوهمي
- النتيجة المتوقعة: الطابعة تطبع الفاتورة (الأولوية محفوظة)، البرنامج يحفظ الطلب في قائمة انتظار، لما السيرفر يرجع يرسل المتراكم.

### السيناريو 5: قياس السرعة
- شغّل الكاشير الوهمي 100 مرة بشكل متتالي
- قس متوسط وقت التمرير للطابعة
- يجب أن يكون أقل من 50ms

## 6. التقنيات والمكتبات المطلوبة

### البرنامج المحلي
- **Node.js** v20+
- **TypeScript** v5+
- **net** (مدمج) - للـ TCP
- **better-sqlite3** - قاعدة البيانات المحلية
- **axios** - الاتصال بالسيرفر
- **pino** - التسجيل

### السيرفر
- **Fastify** - إطار العمل الأساسي
- **@fastify/websocket** - WebSocket
- **@fastify/static** - تقديم الملفات الثابتة

### الواجهات (Display + Staff)
- HTML + CSS + Vanilla JavaScript
- بدون أي framework (لتقليل التعقيد بهذه المرحلة)

### أدوات التطوير
- **ts-node** أو **tsx** - تشغيل TypeScript مباشرة
- **nodemon** - إعادة تشغيل تلقائية أثناء التطوير

## 7. معايير القبول (Acceptance Criteria)

النسخة التجريبية تعتبر ناجحة إذا حققت كل النقاط التالية:

- [ ] الكاشير الوهمي يبعت بيانات للبرنامج المحلي بنجاح
- [ ] البرنامج المحلي يستقبل البيانات ويمررها للطابعة الوهمية في أقل من 50ms
- [ ] الطابعة الوهمية تعرض الفاتورة بشكل صحيح
- [ ] البرنامج المحلي يستخرج رقم الطلب بنجاح من النص
- [ ] الرقم يظهر على شاشة العرض في قسم "قيد التحضير" خلال ثانية واحدة
- [ ] الموظف يقدر يضغط "جاهز" من صفحته، الرقم ينتقل لـ "جاهز للاستلام" مع صوت تنبيه
- [ ] النظام يصمد لـ 100 طلب متتالي بدون أخطاء
- [ ] إذا الطابعة مفصولة، البرنامج لا يقع، بل يعيد المحاولة
- [ ] إذا السيرفر مقفل، البرنامج يحفظ الطلبات ويرسلها لما يرجع

## 8. المخاطر والافتراضات

### الافتراضات
- جهاز التطوير يدعم تشغيل 4-5 عمليات Node.js متوازية بدون مشاكل
- escpresso أو بديله الوهمي يشتغل بشكل صحيح
- صلاحيات ويندوز تسمح بفتح بورتات على localhost

### المخاطر
| الخطر | الاحتمال | التأثير | التخفيف |
|------|---------|---------|---------|
| escpresso صعب التشغيل على ويندوز | متوسط | منخفض | استخدام بديل بسيط بـ Node.js |
| استخراج رقم الطلب بـ regex فشل | منخفض | متوسط | الاعتماد على رقم تسلسلي بديل |

## 9. الخطوات التالية بعد هذه المرحلة

عند إكمال هذه المرحلة بنجاح:

1. **شراء طابعة حرارية شبكية حقيقية** (50-80 دينار) واختبار البرنامج المحلي معها
2. **تنزيل برنامج كاشير حقيقي** (مثل Loyverse POS) واختبار التكامل الكامل
3. **بناء السيرفر السحابي** بدلاً من السيرفر المحلي
4. **إضافة لوحة الإعدادات** لصاحب المطعم
5. **تحويل البرنامج المحلي إلى Windows Service**
6. **اختبار في مطعم تجريبي حقيقي**
