# PRD #6: لوحة التحكم السحابية (Admin Panel)

> **الغرض من هذا المستند:** تحديد متطلبات لوحة التحكم التي يستخدمها صاحب المشروع لإدارة المطاعم وإعداداتها ومراقبة حالة البرامج المحلية.

> **المرجعيات:**
> - PRD #3: قسم ٣ (قاعدة البيانات)، قسم ٥ (REST API)
> - `decisions_session_05.md`: قسم ٣.٥ (PIN)
> - PRD #1: قسم ٤.١ (شكل config.json)

> **التبعيات:** PRD #3 مكتمل (قاعدة البيانات والـ endpoints الأساسية موجودة).

---

## ١. النطاق

### ١.١ ما يشمله هذا الـ PRD

- مصادقة الأدمن (تسجيل دخول)
- إدارة المطاعم (إنشاء، تعديل، تعطيل)
- تعديل إعدادات كل مطعم (`config` و `staff_pin`)
- مراقبة حالة البرامج المحلية (متصل/غير متصل، lan_ip، آخر ظهور)
- مراقبة الطلبات النشطة لكل مطعم
- إصدار `settings_updated` يدوياً عند الحاجة
- عرض بسيط لسجل الأحداث الأخيرة (event_log)

### ١.٢ ما هو خارج النطاق

- **إدارة متعددة المستخدمين بصلاحيات متعددة** (نسخة 1.0: حساب أدمن واحد فقط)
- **وصول صاحب المطعم للوحة** (نسخة 1.0: صاحب المشروع يدير كل شيء بالنيابة)
- **التحليلات المتقدمة** (لوحات بيانية، KPIs، إلخ — مؤجَّل)
- **الفوترة والاشتراكات** (مؤجَّل)
- **سجلات تدقيق متقدمة** (audit logs مفصّلة، نكتفي بالـ event_log البسيط)

### ١.٣ المبادئ الموجِّهة

- **بساطة قبل الجمال:** الأدمن صاحب المشروع، يحتاج أداة وظيفية لا تطبيق جذّاب
- **المسؤوليات بدلاً من الميزات:** كل صفحة تخدم مهمة محددة، لا shortcuts ولا miracles
- **الأمان عبر البساطة:** حساب واحد، بدون تعقيدات الصلاحيات

## ٢. المستخدمون والصلاحيات

### ٢.١ الحساب الوحيد للنسخة 1.0

**حساب أدمن واحد** فقط (صاحب المشروع).

```sql
CREATE TABLE admin_users (
  id              SERIAL PRIMARY KEY,
  username        VARCHAR(50) NOT NULL UNIQUE,
  password_hash   VARCHAR(200) NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_login_at   TIMESTAMPTZ,
  is_active       BOOLEAN NOT NULL DEFAULT true
);
```

في النسخة 1.0، يُنشَأ سطر واحد عند إعداد السيرفر (عبر CLI script أو env variables).

### ٢.٢ التطوير المستقبلي

في نسخ لاحقة:
- إضافة حسابات أدمن إضافية
- إضافة دور "مدير مطعم" (يصل لمطعمه فقط)
- إضافة سجل تدقيق (من فعل ماذا ومتى)

التصميم يجب أن يكون **مرناً** لإضافة هذه الأدوار لاحقاً، لكن لا نطبّقها الآن.

## ٣. مصادقة الأدمن

### ٣.١ تسجيل الدخول

**`POST /admin/api/login`**

**Request:**
```json
{
  "username": "owner",
  "password": "..."
}
```

**Response (200):**
```json
{
  "token": "jwt-token-here",
  "expires_at": 1746345600000,
  "user": { "username": "owner" }
}
```

**Response (401):**
```json
{ "error": "invalid_credentials" }
```

### ٣.٢ JWT للأدمن

**سبب استخدام JWT هنا (بعكس صفحة الموظف):**
- صاحب المشروع يستخدم اللوحة بنفسه فقط
- لا حاجة لطرد فوري عند تغيير كلمة السر
- JWT أقل تعقيداً من إرسال username/password كل request

```
Token expires_at: 8 ساعات
Refresh: عند انتهاء الصلاحية → إعادة تسجيل دخول
```

### ٣.٣ Password Hashing

```javascript
const bcrypt = require('bcrypt')
const hash = await bcrypt.hash(password, 12)  // 12 rounds
```

### ٣.٤ Rate Limiting على Login

5 محاولات في الدقيقة لكل IP. بعدها 15 دقيقة حظر.

### ٣.٥ Middleware للحماية

```javascript
async function requireAdmin(request, reply) {
  const auth = request.headers.authorization
  if (!auth?.startsWith('Bearer ')) {
    return reply.code(401).send({ error: 'unauthorized' })
  }
  
  const token = auth.replace('Bearer ', '')
  try {
    const decoded = jwt.verify(token, JWT_SECRET)
    request.admin = decoded
  } catch (err) {
    return reply.code(401).send({ error: 'invalid_token' })
  }
}
```

كل endpoints `/admin/api/*` (ما عدا `/login`) محمية بهذا الـ middleware.

## ٥. الصفحة الرئيسية (Dashboard)

### ٥.١ المحتوى

**عدّادات بسيطة:**
- إجمالي المطاعم النشطة
- المطاعم المتصلة الآن (عبر WebSocket)
- إجمالي الطلبات النشطة (preparing + ready) عبر كل المطاعم
- إجمالي الأحداث في آخر 24 ساعة

**قائمة "تحتاج انتباه":**
- مطاعم نشطة لكن برنامجها المحلي غير متصل > 5 دقائق
- مطاعم لم تُسجِّل أي حدث > 24 ساعة (ربما متوقفة)

**قائمة "آخر النشاطات":**
- آخر 10 أحداث في `event_log` عبر كل المطاعم
- اسم المطعم + نوع الحدث + الوقت

### ٥.٢ تحديث البيانات

- تحديث تلقائي كل 30 ثانية (polling)
- لا حاجة لـ WebSocket في اللوحة (مستخدم واحد، حركة قليلة)

## ١١. تعطيل/تفعيل مطعم

### ١١.١ تعطيل

**`POST /admin/api/restaurants/{id}/deactivate`**

**النتيجة:**
- `is_active = false` في DB
- البرنامج المحلي يُرفَض عند المصادقة (401)
- الموظفون لا يستطيعون الدخول
- الشاشة تتوقف عن استقبال أحداث

**التأكيد:**

```
هل أنت متأكد من تعطيل مطعم الشرق؟
- البرنامج المحلي سيتوقف عن الاتصال
- الشاشة ستتوقف
- الموظفون لن يستطيعوا الدخول

البيانات تبقى محفوظة. يمكن إعادة التفعيل لاحقاً.

[تعطيل]  [إلغاء]
```

### ١١.٢ التفعيل

`is_active = true`. كل شيء يعمل من جديد.

### ١١.٣ ملاحظة على الحذف

**النسخة 1.0: لا حذف نهائي للمطاعم.** فقط تعطيل.

السبب: الحذف خطر (يفقد الأرشيف). التعطيل يحقق نفس الغرض بدون مخاطرة.

### ١٢.٢ Frontend

**خياران:**

**أ. Server-rendered HTML + minimal JS (موصى به للبساطة)**
- Fastify يخدم HTML templates (بـ Handlebars أو EJS)
- JavaScript بسيط للنماذج (form validation، AJAX)
- لا framework
- **مناسب لـ:** أدمن صاحب المشروع، استخدام محدود
- **حجم:** صغير، سريع

**ب. SPA منفصل (Preact أو React)**
- جذاب أكثر، تجربة أفضل
- يحتاج build step
- **مناسب لـ:** نسخ مستقبلية مع عدة أدمن

**رأيي الصريح:** اذهب للخيار (أ) في النسخة 1.0. اللوحة أداة لك أنت، لا تحتاج تعقيد SPA.

### ١٢.٣ التصميم

- استخدم CSS framework خفيف: **Tailwind** (CDN) أو **Pico CSS** (5 KB)
- لا تضيع وقتاً في التصميم — اللوحة وظيفية، ليست تسويقية

## ١٣. الأمان

### ١٣.١ HTTPS فقط

كل الاتصالات بلوحة التحكم عبر HTTPS.

### ١٣.٢ كلمة سر قوية

- 12+ حرف
- خليط من أحرف وأرقام ورموز
- التحقق عند التسجيل

### ١٣.٣ JWT Secret

- secret طويل (64+ حرف عشوائي)
- في environment variable
- لا في الكود

### ١٣.٤ CSRF

- استخدام `SameSite=Strict` على cookies
- CSRF token على forms حساسة (تغيير PIN، تعطيل مطعم)

### ١٣.٥ Rate Limiting

- Login: 5 محاولات / دقيقة
- API الأدمن: 100 request / دقيقة (سخي)

## ١٦. ملاحظات إضافية

### ١٦.١ النسخة 2.0 — تطورات متوقعة

- حسابات متعددة بصلاحيات (super_admin, admin, restaurant_manager)
- وصول صاحب المطعم للوحة الخاصة به
- تحليلات (متوسط زمن التحضير، عدد الطلبات اليومية، وقت الذروة)
- Email notifications للأدمن (مطعم متوقف > 30 دقيقة، إلخ)
- نسخة احتياطية وتصدير

### ١٦.٢ السلامة

- اللوحة هي **أكبر مفتاح** في النظام. اختراقها = تحكم بكل المطاعم
- بالإضافة لكلمة السر القوية، أنصح:
  - تفعيل **2FA** في النسخة 2.0 (Google Authenticator أو SMS)
  - تسجيل **كل** عمليات الأدمن (audit log)
  - alerting عند تسجيل دخول من IP غير معتاد

في النسخة 1.0، نكتفي بـ HTTPS + كلمة سر قوية + rate limiting.

### ١٦.٣ الأولوية مقارنة بالـ PRDs الأخرى

اللوحة **مهمة لكنها ليست أولوية للإطلاق الأول.**

في النسخة الأولى (تجربة أول مطعم)، يمكنك:
- إنشاء المطعم يدوياً عبر SQL مباشرة
- تعديل الإعدادات يدوياً في DB
- مراقبة الحالة عبر logs

اللوحة تصبح ضرورية عند 3-5 مطاعم. لذا، **يمكن تأجيل تطويرها** بعد إطلاق النسخة 1.0 على المطعم الأول.

**التوصية:** ابنِ النسخة الأولى بدون لوحة تحكم كاملة. اكتفِ بـ scripts بسيطة. أضف اللوحة بعد إثبات الفكرة.
