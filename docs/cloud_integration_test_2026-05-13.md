## What we proved works

1. **Cloud HTTPS** — `curl https://cloud.skrterak.com/health` returns `200 {"status":"ok","timestamp":...}`. nginx fronts it with HSTS, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`.
2. **Cloud WS authentication + nginx upgrade** — handshake on `/ws/local-agent` and `/ws/staff` returns `401` without credentials and `101 Switching Protocols` with a valid `restaurant_id` + `api_key`. Verified with both raw curl and the agent's own `ws` library via `node scripts/setup-helper.js verify-cloud`.
3. **Agent boot against production cloud** — `node src/index.js` with `QM_CONFIG_FILE=config/config.skrterak.json` boots cleanly, validates config, opens cloud WS in ~1.4s, prints the green-check startup banner.
4. **ERPNext → agent → real printer → fallback serial** — once the dev agent was reconfigured with real network values (alias bind on `192.168.1.50:9100`, forward to `192.168.1.190:9100`), ERPNext's print button caused paper to come out, and the agent created a local order via the **fallback serial** path (graceful failure mode — see Problem 5 below).

## Problems we hit and how we resolved them

### 1. Admin panel emits localhost cloud URLs in `config.json`

**Symptom:** First config generated by the admin panel had:
```json
"cloud": {
  "base_url": "http://localhost:3000",
  "ws_url":   "ws://localhost:3000/ws"
}
```
Every restaurant config would be broken in production until fixed.

**Cause:** Config template / generator in the cloud project hardcoded to the dev URL.

**Fix:** User fixed the cloud project to emit production URLs. Re-generated config (`rest_0k6lu7`) came out correct. Verify in future sessions that any new admin-generated config still has `https://cloud.skrterak.com` / `wss://cloud.skrterak.com/ws`.

### 2. Admin panel emits `staff_pin` nested inside `restaurant`

**Symptom:** `setup-helper verify-config` failed:
```
staff_pin: required non-empty string (PRD #8: root-level)
```
because admin panel placed it as `restaurant.staff_pin` while the agent schema (`src/config/schema.js:57-61`) requires it at root.

**Fix:** Same — fixed in cloud project. Re-issued config has `staff_pin` at root.

### 3. Dev machine clock was ~14.5 hours behind real UTC

**Symptom:** `node scripts/setup-helper.js verify-cloud` returned:
```
FAIL_HTTP CERT_NOT_YET_VALID
FAIL_WS   CERT_NOT_YET_VALID
```
while `curl https://cloud.skrterak.com/health` happily returned `200`.

**Cause:** Fresh Let's Encrypt cert had `notBefore = 2026-05-13T12:32:01Z`. Machine clock said `2026-05-12T23:57Z`. Node's bundled OpenSSL strictly compares cert `notBefore` against `Date.now()`; Windows's `curl.exe` uses Schannel and is more lenient about clock skew. The agent rejected the cert as `NOT_YET_VALID`.

**Fix:** `w32tm /resync /force` from an admin PowerShell. After resync, `verify-cloud` returned:
```
OK_HTTP /health (status 200)
OK_WS   /local-agent
```

**Lesson for deployment day:** This will absolutely bite a restaurant cashier PC with clock drift. The current error is cryptic and there is no clue from `curl` that the agent will fail. Action items in `Open issues` below.

### 4. Two-agent routing trap (specific-IP vs wildcard binds on port 9100)

**Symptom:** First ERPNext print after setting up `rest_0k6lu7` did not appear on the staff page — at all.

**Cause:** Both agents were running:

| PID | Bind | Restaurant | Cloud |
|---|---|---|---|
| installed service | `192.168.1.50:9100` (specific IP via alias) | `rest_knk8aa` | `http://192.168.1.249:3000` ❌ dead local mock |
| dev agent | `0.0.0.0:9100` (wildcard) | `rest_0k6lu7` (Q1) | `https://cloud.skrterak.com` ✅ |

Windows routes incoming TCP connections to the **most specific** bind. ERPNext was configured to print to `192.168.1.50:9100` (Windows printer port redirected from 2026-05-12), so bytes went to the installed service → tried to publish to a dead cloud → event never reached `cloud.skrterak.com` → nothing on staff page.

**Fix:** User stopped the installed service (`Stop-Service QueueManager` from admin shell). Dev agent restarted with real network values:
```json
"network": {
  "cashier_ip": "192.168.1.50",        // same-machine simulation per 2026-05-12 §4
  "printer_old_ip": "192.168.1.50",    // intercept bind
  "printer_new_ip": "192.168.1.190",   // forward target = real printer
  "printer_port": 9100,
  "interface_name": "Ethernet 2"       // where the alias already lives
}
```
After restart, agent bound `192.168.1.50:9100` directly (specific IP from `printer_old_ip`) and the cloud WS connected for the right restaurant.

### 5. ERPNext via Windows driver still emits raster bytes — extractor falls back to local serial

**Symptom (predicted from `printer_integration_test_2026-05-12.md` §5, confirmed today):** ERPNext print produced paper output, but the byte stream the agent received was 32982 bytes starting with:
```
ESC @  GS U B  ...  GS v 0  H 00 18 00  [~32KB of bitmap]
```
That's `ESC @` (init) + `GS v 0` (raster bit image command) followed by the bitmap of the entire receipt. The literal text `رقم الطلب: 42` is not in the byte stream.

The regex `رقم الطلب:?\s*(\d+)` finds no match. The agent logged:
```
[WARN] (extractor) order number extraction failed — using local serial {"serial":1, ...}
[INFO] new order #1 (fallback) {"encoding":null,"bytes":32982}
```

**Behavior note (this is new info):** the agent has a **graceful fallback** for extraction failure — it assigns an incrementing local serial counter instead of dropping the event. So the staff page should still receive an order, just with a wrong number that doesn't match the receipt the customer is holding.

This is acceptable as a smoke-test signal ("data does flow end-to-end") but unacceptable as a production behavior — the customer's order number on paper will never match the screen.

**Fix options (unchanged from 2026-05-12):**
- Switch the Windows printer to "Generic / Text Only" driver → emits plain text + minimal ESC/POS → regex matches.
- Drive ERPNext from a raw-ESC/POS print format (custom Print Format pointed at `192.168.1.50:9100`) bypassing the Windows spooler.
- Replace ERPNext with a POS that prints raw ESC/POS to TCP:9100 (most Arabic standalone POS does this — the supported path).

## Architecture clarifications learned today

- **Production cloud WS paths:** `${cloud.ws_url}/local-agent` (agent) and `${cloud.ws_url}/staff` (staff page). Set `cloud.ws_url = "wss://cloud.skrterak.com/ws"`; the agent code in `src/core/sync/client.js:112` and `src/index.js:110` appends the channel name.
- **`setup-helper verify-cloud` is the right diagnostic.** It uses the agent's own TLS stack and `ws` library, so it catches things `curl` can't (e.g. clock skew → CERT_NOT_YET_VALID). Use it instead of curl when validating production cloud reachability from a dev/install machine.
- **Bind address note.** When `printer_old_ip` is empty, the agent binds the intercept on `0.0.0.0:printer_port` (wildcard) — captures all interfaces. When `printer_old_ip` is set, it binds on that specific IP. Specific-IP binds beat wildcard binds for connection routing on Windows.
- **Extraction fallback.** `src/core/extractor.js` (or wherever the regex lives) returns a sentinel that triggers an incrementing local-serial path in the order creation flow. Confirmed today, not previously documented.
